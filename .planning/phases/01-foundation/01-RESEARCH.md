## Summary

Phase 1 builds the project from zero: scaffold → design tokens → layout shell → SEO wiring → Vercel pipeline. No content sections are produced. The output is a deployed shell page with a sticky header, empty `<main>`, footer, correct color/typography tokens, and all SEO meta tags present in the HTML source.

The stack is well-understood and the integration paths are verified against official documentation. The two most important non-obvious facts are: (1) Astro 5.x latest is `5.18.1` — much newer than the `5.8` referenced in earlier research — but the `^5` constraint still applies; (2) `@astrojs/sitemap` does NOT generate `robots.txt` — that file must be placed manually in `public/` to satisfy SEO-06.

The research establishes a clear, prescriptive implementation path. No exploratory choices remain — all stack decisions are locked.

**Primary recommendation:** Scaffold with `npm create astro@latest`, pin all packages at verified current versions, implement `robots.txt` as a static `public/` file, and connect Vercel via GitHub Git integration before writing a single component.

## Architecture Patterns

### Recommended Project Structure

```
src/
├── components/
│   └── layout/
│       ├── SiteHeader.astro     # Sticky header — logo + nav links
│       └── SiteFooter.astro     # Footer — copyright line
├── layouts/
│   └── Layout.astro             # HTML shell, <head> with SEO, slot
├── lib/
│   └── metadata.ts              # SITE_URL, SITE_TITLE, SITE_DESCRIPTION exports
├── pages/
│   └── index.astro              # Minimal shell — Layout + Header + main + Footer
└── styles/
    └── global.css               # @import "tailwindcss" + @theme {} tokens + base reset

public/
└── robots.txt                   # Static robots.txt with sitemap URL reference

tsconfig.json                    # Extends astro/tsconfigs/strict
```

### Pattern 1: Tailwind v4 CSS-First Configuration

**What:** Design tokens live in `@theme {}` blocks inside `global.css`. No `tailwind.config.js` exists. Tailwind v4 reads the CSS file directly via the Vite plugin.

**When to use:** Always — this is the v4 required pattern. The old `tailwind.config.js` approach is unsupported in v4.

**Example:**
```css
/* src/styles/global.css */
/* Source: https://tailwindcss.com/docs/installation/framework-guides/astro */
@import "tailwindcss";

@theme {
  --color-surface: #0a0a0a;
  --color-surface-raised: #111111;
  --color-text-primary: #f5f5f5;
  --color-text-muted: #a1a1a1;
  --color-border: #1f1f1f;
  --color-accent: #ffffff;

  --font-family-sans: 'Geist Sans', ui-sans-serif, system-ui, sans-serif;
}

/* Base reset */
html {
  scroll-behavior: smooth;
  background-color: var(--color-surface);
  color: var(--color-text-primary);
  font-family: var(--font-family-sans);
}

body {
  min-height: 100vh;
}
```

**Vite plugin in `astro.config.mts`:**
```typescript
// Source: https://tailwindcss.com/docs/installation/framework-guides/astro
import { defineConfig } from 'astro/config';
import tailwindcss from '@tailwindcss/vite';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  site: 'https://apexcode.vercel.app',
  integrations: [sitemap()],
  vite: {
    plugins: [tailwindcss()],
  },
});
```

### Pattern 2: SEO Component in Layout

**What:** `astro-seo`'s `<SEO>` component is used inside `Layout.astro`'s `<head>`. Props are passed from `index.astro` via the layout's Props interface. Canonical metadata values come from `metadata.ts` exports.

**When to use:** All pages — the layout is the single SEO injection point.

**Example:**
```astro
---
// src/layouts/Layout.astro
// Source: https://github.com/jonasmerlin/astro-seo
import { SEO } from 'astro-seo';
import '@fontsource/geist-sans/400.css';
import '@fontsource/geist-sans/600.css';
import '../styles/global.css';
import { SITE_URL, SITE_TITLE, SITE_DESCRIPTION } from '../lib/metadata';

interface Props {
  title?: string;
  description?: string;
}

const {
  title = SITE_TITLE,
  description = SITE_DESCRIPTION,
} = Astro.props;
---
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <SEO
      title={title}
      description={description}
      canonical={SITE_URL + '/'}
      openGraph={{
        basic: {
          title: title,
          type: 'website',
          image: `${SITE_URL}/og-image.png`,
          url: SITE_URL + '/',
        },
        optional: {
          description: description,
          siteName: 'Apex Code',
        },
      }}
    />
  </head>
  <body class="bg-surface text-text-primary">
    <a href="#main-content" class="sr-only focus:not-sr-only">Skip to main content</a>
    <slot />
  </body>
</html>
```

### Pattern 4: Font Loading (Fontsource)

**What:** Import weight-specific CSS files inside `Layout.astro` frontmatter. Only import the weights you use (400, 600). This keeps the font payload minimal.

```typescript
// Inside Layout.astro frontmatter
import '@fontsource/geist-sans/400.css';
import '@fontsource/geist-sans/600.css';
```

**Why not `import '@fontsource/geist-sans'` (default):** The default import loads weight 400 only. Explicit imports make it clear which weights are bundled.

### Pattern 5: Static robots.txt

**What:** Plain text file in `public/` — Astro copies `public/` contents verbatim to the build output root. No JS endpoint needed.

```
# public/robots.txt
User-agent: *
Allow: /
Sitemap: https://apexcode.vercel.app/sitemap-index.xml
```

**Critical detail:** `@astrojs/sitemap` outputs `sitemap-index.xml` as the root index file (not `sitemap.xml`). The robots.txt must reference `sitemap-index.xml`, not `sitemap.xml`.

### Pattern 6: TypeScript strict tsconfig

**What:** Generated by `npm create astro@latest` when "TypeScript strict" preset is selected.

```json
{
  "extends": "astro/tsconfigs/strict",
  "include": [".astro/types.d.ts", "**/*"],
  "exclude": ["dist"]
}
```

No modifications needed. `astro/tsconfigs/strict` enables `strict: true`, `noUncheckedIndexedAccess`, and Astro-specific type support.

### Anti-Patterns to Avoid

- **Using `tailwind.config.js`:** Not supported in Tailwind v4. All configuration must live in `@theme {}` blocks in CSS.
- **Installing `@astrojs/vercel` adapter:** The SSR adapter breaks static output. Do not install for a static site.
- **Using `@astrojs/react` or any island adapter:** Phase 1 has zero interactive components. Adding React reintroduces runtime JS cost.
- **Google Fonts `<link>` tag:** Adds render-blocking external request. Use `@fontsource/geist-sans` only.
- **Importing full `@fontsource/geist-sans` without weight specifiers:** The package supports 9 weights (100–900). Only import 400 and 600 to minimize font payload.
- **Hardcoding site URL in multiple places:** All SEO strings originate from `src/lib/metadata.ts`. Phase 3 domain swap is a one-file change.
- **Referencing `sitemap.xml` in robots.txt instead of `sitemap-index.xml`:** `@astrojs/sitemap` generates an index file at `sitemap-index.xml`. Pointing robots.txt at `sitemap.xml` will 404.
- **Using `astro.config.mjs` extension:** Project rule is TypeScript only. Use `astro.config.mts` for the TypeScript ESM config file.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OG meta tags, canonical, twitter card | Custom `<head>` template | `astro-seo` | Missing tags are invisible at dev time but break social sharing. `astro-seo` forces completeness via TypeScript prop types. |
| Sitemap generation | Manual sitemap builder | `@astrojs/sitemap` | Sitemap content must match build output. Manual files go stale. The integration reads Astro's route manifest directly. |
| Font self-hosting | Download and serve font files manually | `@fontsource/geist-sans` | Fontsource handles WOFF2 format selection, weight subsetting, and cache headers. Manual hosting requires build pipeline work. |
| CSS reset | Hand-written reset | Tailwind's built-in Preflight (included via `@import "tailwindcss"`) | Tailwind v4 Preflight normalizes browser defaults automatically. |

**Key insight:** This stack's "don't hand-roll" surface is small because Astro does most of the heavy lifting at build time. The main trap is manual SEO tags — they look complete but inevitably have gaps.

## Common Pitfalls

### Pitfall 1: Wrong sitemap file reference in robots.txt

**What goes wrong:** `robots.txt` points to `sitemap.xml` but `@astrojs/sitemap` generates `sitemap-index.xml`. The sitemap is present but crawlers cannot find it via `robots.txt`.

**Why it happens:** The integration changed its output filename; most tutorials predate this change.

**How to avoid:** Always reference `sitemap-index.xml` in `robots.txt`.

**Warning signs:** `curl https://your-site.vercel.app/sitemap.xml` returns 404 while `sitemap-index.xml` returns 200.

### Pitfall 2: `@astrojs/vercel` adapter installed on a static site

**What goes wrong:** Build output switches from static files to serverless functions. Vercel costs money for SSR invocations; static is free on Hobby plan. Bundle size increases.

**Why it happens:** Easy to confuse "deploy to Vercel" with "install Vercel adapter."

**How to avoid:** Never run `astro add vercel`. Static Astro sites deploy to Vercel with zero adapter.

**Warning signs:** `astro.config.mts` has `import vercel from '@astrojs/vercel'` and `output: 'server'`.

### Pitfall 3: Absolute OG image URL missing base URL

**What goes wrong:** `og:image` renders as `/og-image.png` (relative path). Social media crawlers require absolute URLs — the card silently fails to show the image.

**Why it happens:** Astro's `Astro.url` gives absolute URL, but if the value is assembled by string concatenation incorrectly, the base URL is lost.

**How to avoid:** `og:image` value must be `${SITE_URL}/og-image.png`, where `SITE_URL` includes the scheme (`https://`). Verify in `view-source` that the rendered tag is an absolute URL.

**Warning signs:** `<meta property="og:image" content="/og-image.png">` (no domain prefix) in the page source.

### Pitfall 4: `tailwind.config.js` created alongside v4

**What goes wrong:** Tailwind v4 ignores `tailwind.config.js`. Design tokens defined there silently have no effect. Custom colors don't apply.

**Why it happens:** Developer habit from Tailwind v3.

**How to avoid:** Do not create `tailwind.config.js`. All tokens go in `@theme {}` blocks in `global.css`.

**Warning signs:** Color classes like `bg-surface` render as transparent or the wrong color.

### Pitfall 5: Missing `site` property in `astro.config.mts`

**What goes wrong:** `@astrojs/sitemap` skips sitemap generation entirely and logs a warning. `sitemap-index.xml` is absent from the build output.

**Why it happens:** `site` is optional from Astro's perspective but required for sitemap.

**How to avoid:** Set `site: 'https://apexcode.vercel.app'` in `astro.config.mts` before the first build.

**Warning signs:** Build output has no `sitemap-index.xml` or `sitemap-0.xml` files.

### Pitfall 6: Vercel Hobby plan commercial use restriction

**What goes wrong:** Hobby plan Terms of Service prohibit commercial use. Apex Code is a commercial studio — the site monetizes the founders' services.

**Why it matters for Phase 1:** The Phase 1 deploy goes to `.vercel.app` on whatever plan is active. If the account is on Hobby, the site is technically ToS-violating from the first deploy.

**How to avoid:** Flag this for the human owner before connecting Vercel. Options: (1) Upgrade Vercel to Pro ($20/month), or (2) switch to Cloudflare Pages (free tier has no commercial restriction). Phase 1 proceeds with Vercel; Phase 3 is the deadline for resolving this.

**Warning signs:** `STATE.md` already flags this as a known blocker.

### Pitfall 7: Nav link anchor IDs defined in Phase 2, referenced in Phase 1

**What goes wrong:** `SiteHeader.astro` links to `#services`, `#why-us`, `#process`, `#contact` — but those `id` attributes don't exist until Phase 2. Clicking nav links in Phase 1 scrolls nowhere.

**Why it happens:** Phase boundary: header is Phase 1, section content is Phase 2.

**How to avoid:** This is expected behavior in Phase 1. Document it in the component. Add a code comment: `{/* Section IDs created in Phase 2 */}`. Do not defer the header build — the anchor hrefs are correct, just orphaned until Phase 2.

**Warning signs:** None in Phase 1 (expected). Flag if anchor links are still broken after Phase 2 completes.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tailwind.config.js` + `@tailwind base/components/utilities` directives | `@theme {}` blocks in CSS + `@import "tailwindcss"` | Tailwind v4 (2025) | No JS config file; configuration is pure CSS |
| `@astrojs/tailwind` integration | `@tailwindcss/vite` Vite plugin | Tailwind v4 / Astro 5.2+ | No separate Astro integration needed; plugin goes directly in `vite.plugins` |
| `tailwind.config.js` for `content:` paths | Auto-detection via Vite plugin | Tailwind v4 | No manual content path configuration required |
| Astro 4.x | Astro 5.x (5.18.1 as of 2026-03-26) | Dec 2024 | Content Collections API v2, improved TypeScript, Session API; no breaking changes for static sites |
| `astro.config.mjs` | `astro.config.mts` | Astro 5.x recommendation | TypeScript config file is fully supported and matches project TypeScript-only rule |

**Deprecated/outdated:**
- `@astrojs/tailwind` integration package: deprecated — use `@tailwindcss/vite` Vite plugin instead for v4
- `tailwind.config.js`: not supported in Tailwind v4 projects
- Astro `legacy` tag (4.16.x): still on npm but 5.x is the active line
